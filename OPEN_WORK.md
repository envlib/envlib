## Backlog

- [ ] **[cfdb] `EDataset.push()` inside a creation/modification session uploads a stale SysMeta** (found 2026-07-02): the full `SysMeta` (variables, shapes) reaches the booklet metadata slot only at close (`dataset_finalizer`, `utils.py:182-190`); `EDataset.__init__` writes an EMPTY SysMeta at open (`edataset.py:26`). A `ds.push()` before close therefore publishes a remote whose metadata has no/stale variables — readers see an empty or shape-stale dataset. The documented close-then-reopen-push workflow avoids it, but programmatic callers (envlib `cat.publish()`) will trip on it. Fix shape: `EDataset.push()` syncs the current `_sys_meta` via `set_metadata()` before `changes().push()`. Until then, envlib must implement close→reopen→push internally.
- [ ] **[cfdb] `open_edataset(flag='w')` with a fresh local file treats an EXISTING remote dataset as create-new** (found 2026-07-02 during the append smoke): `edataset.py:125` sets `create=True` from local-file absence alone; `Dataset.__init__` then builds an empty `SysMeta` (`main.py:762`) instead of loading the remote's (`main.py:776`), and `dataset_finalizer` (`utils.py:182-190`) writes that empty SysMeta over the real one on close — the next push would destroy the remote dataset's structure metadata. Fix shape: decide `create` from remote existence too (e.g., `flag == 'n'` or (local absent AND remote metadata absent)). Workaround until fixed: hydrate with a read open first, then reopen `'w'` — envlib's `cat.register()`/append flows must use this pattern.

- [x] ~~**[s3func] Conditional-write (CAS) probe results across providers**~~ — **CLOSED 2026-07-03**: findings now recorded in-repo at `s3func/benchmarks/results_visibility_lag.md` (with the probe as the allowlist qualification gate). Original record: (2026-07-02, live probes; informs the lock design): **B2** rejects `If-None-Match` PUT with loud `NotImplemented` (S3-compat) and the native API has no conditional capability either (spec-checked). **MEGA S4** documents and honors `If-None-Match`/`If-Match` PUT *sequentially* (412 + body preserved) but is **NOT atomic under concurrency** — 8 simultaneous `If-None-Match:*` creates on one fresh key ALL returned 200 in 4 of 5 rounds (check-then-write, not CAS; probe script in session scratchpad, ready as a MEGA bug report reproducer). Design consequence: hardened bakery is the primary lock everywhere; a provider-allowlist CAS tier is cheap scaffolding but currently has zero qualifying members (AWS proper documents atomic conditional writes; unverified here). S4's conditional headers remain useful as non-atomic stale-write narrowing (e.g., If-Match on index pushes), never as a mutex. Also decided (Mike): deprecating s3func's native-B2 module is on the table pending the LIST-vs-native-version-list visibility measurement.
- [x] ~~**[s3func] Fix bakery-lock weaknesses**~~ — **RESOLVED 2026-07-03** (see the 0.9.0 review+release item above: root-caused via instrumented campaign, fixed, validated 0-violations). Original record:
  1. *Accumulation bug* (by inspection, `s3func/locking.py:175-185`): `locked` is reassigned per iteration in `_check_for_older_objs`, so only the last-listed contender counts — with 3+ simultaneous writers, two can both acquire. One-line fix (`locked = locked or ...` or early-return).
  2. *LIST-after-PUT visibility race* (observed live): `acquire()` PUTs its lock objects then immediately LISTs for competitors; a competitor whose lock PUT hasn't propagated into LIST results yet is invisible, so two writers can both acquire even with only 2 contenders. Observed as a flaky lost-update in a threaded two-writer test against B2 (on ebooklet code with the 0.8.4 fixes — the lock itself failed to serialize). Mitigation shape: self-visibility check (poll LIST until your own lock objects appear, then evaluate competitors) + settle-delay re-list; reproducer: looped two-process acquire with mutual-exclusion assertion under parallel load. Corroboration: Mike's own s3func lock tests historically showed rare, irreproducible random failures — consistent with this window. Until fixed, truly-simultaneous first-contact writers to the same remote remain a small race window; ebooklet 0.8.4's post-lock metadata refresh protects every case where the lock actually serializes.
